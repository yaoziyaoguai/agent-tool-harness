## 二、为什么 v2.0 之后不能直接进入大功能

1. **文档体系不完整**：产品意图、用户场景、项目约束散落在 README、ROADMAP、ARCHITECTURE 各处，没有独立文档。后续 Coding Agent 或新贡献者缺乏明确的"什么能做、什么不能做"的依据。
2. **v2.0 之后方向未定**：ROADMAP 说"主线只到 v2.0"，但 v3.0 backlog 只有否定形式的"不包含"表，没有正面规划。
3. **缺少真实用户反馈**：v2.0 设计为"内部小团队试用"，但如果没有真实试用反馈，v3.0 的能力设计可能就是闭门造车——做了一堆没人用的功能。
4. **版本号不一致**：`__version__ = "0.1.0"` 与 git tag `v2.0` 矛盾，需要在后续统一。

## 四、三个可能的方向

### 方向 A：继续等待反馈 + 工程质量打磨

**进入条件**：
- 当前 0 份真实非维护者反馈
- maintainer 判断"需要先让内部团队试用，收集反馈再决定"

**允许做的事**：
- 工程质量改进（不改变 product surface）
  - 统一版本号（`__version__` / `pyproject.toml` / git tag）
  - 添加 CI 配置（GitHub Actions）
  - 添加代码覆盖率工具
  - 优化运行速度（不改功能）
- 文档维护（修正错误、补充遗漏）
- 等待反馈期间不新增功能代码

**停止条件**：≥1 份真实反馈到达 → 分流决策

### 方向 B：v2.x Internal Trial Patch

**进入条件**：
- 收到 ≥1 份真实非维护者反馈
- 反馈经 `FEEDBACK_TRIAGE_WORKFLOW.md` 分流为 `v2.x patch`

**允许做的事**：
- 修复反馈中指出的 bug
- 改进 CLI 体验（如更好的错误提示）
- 修复文档断点
- 补测试覆盖 gap

**禁止做的事**：
- ❌ 引入 v3.0 能力（真实 LLM judge / MCP executor / Web UI）
- ❌ 大规模重构
- ❌ 新增依赖

**停止条件**：
- 反馈中指出的问题全部修复
- 或反馈积累到 ≥3 份 → 触发 v3.0 gate 评估

### 方向 C：v3.0 Planning

**进入条件**（来自 `ROADMAP.md` v3.0 gate）：
- **≥3 份真实非维护者反馈**（maintainer rehearsal 和 synthetic 不计入）
- 至少 1 份反馈来自真实内部团队的端到端试用
- 反馈中不包含 security-blocker（如有，先处置）

**允许做的事**：
- 编写 v3.0 PRD（产品需求文档）
- 编写 v3.0 技术设计文档
- 原型验证（prototype，不合入 main）
- 评估真实 LLM judge 的可行性和风险

**禁止做的事**（在 planning 完成并 Review 通过之前）：
- ❌ 合入任何 v3.0 功能代码到 main
- ❌ 接真实 LLM API
- ❌ 实现 MCP executor
- ❌ 开发 Web UI

**停止条件**：
- v3.0 PRD + 技术设计通过 Review
- 或 planning 过程中发现 v3.0 的某个能力在当前约束下不可行 → 调整范围

## 五、决策输入

决定进入哪个方向之前，maintainer 需要确认以下输入：

### 必须确认
1. **当前真实非维护者反馈数**：是 0 / 1 / 2 / 3+？
2. **是否有内部团队表示愿意试用**：如果有，何时可以开始？
3. **Documentation Consolidation 的产出质量**：独立 Agent 审计是否通过？

### 建议确认
4. **公司层面对 AI Agent 工具的投入程度**：如果公司正在大力投入 Agent，v3.0 的时间窗口可能需要提前
5. **是否有已知的"如果不修就会阻塞试用"的问题**：如果有，应优先 v2.x patch
6. **版本号策略**：统一为 `2.0.0` 还是重新审视整个版本策略？

## 六、独立 Agent 审计要看什么

当本轮文档补齐完成后，建议启动一个独立 Agent 对文档体系做审计。以下是建议的审计维度：

1. **完整性**：16 份文档是否全部存在、章节是否完整
2. **一致性**：文档之间是否有矛盾（特别是产品意图 vs 约束 vs 架构）
3. **可追溯性**：每个判断是否有项目文件证据（不凭空断言）
4. **可执行性**：文档是否足够具体到可以指导 Coding Agent 实现（不是空泛模板）
5. **导航性**：新读者能否通过 INDEX.md 在 1 分钟内找到需要的文档
6. **历史文档处理**：STALE_DOCS_AUDIT.md 的分类是否合理

审计结果应形成一份结构化报告，列出具体问题（含文档路径和行号），供人工 Review 修正。
